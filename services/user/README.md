User service module providing CRUD operations, role management, RBAC authorization helpers, and password authentication for lt_users. Types and constants live in `types.ts`, database CRUD in `crud.ts`, role queries in `roles.ts`, role-based access checks in `rbac.ts`, work-surface scope in `scope.ts`, and password verification in `auth.ts`. The barrel `index.ts` re-exports the public API.

A role grant carries a management `type` (member/admin/superadmin) plus two work-surface scope axes — `read_scope` (self|all) and `write_scope` (none|self|all) — that gate the task-queue verbs (search / claim / ack / delete). `scope.ts` holds the `effectiveScope` resolver, which forces admin/superadmin to act on the whole queue; `rbac.ts` exposes `getRoleScope`/`getRoleWriteScope` for per-role enforcement.

All static SQL is centralized in `sql.ts` and imported as named constants. The `lt_user_roles` join table uses a composite primary key on `(user_id, role)`, which the `UPSERT_USER_ROLE` query leverages with `ON CONFLICT (user_id, role) DO UPDATE SET type, read_scope, write_scope` for atomic role-type and scope promotion. `attachRolesToMany` uses `ANY($1)` for batch role loading to avoid N+1 queries when listing users. The dynamic `UPDATE` built in `updateUser` and the dynamic `SELECT` in `listUsers` remain inline because their SQL is constructed at runtime, but all parameters are positionally bound to prevent injection.
