Role management service — maintains the `lt_roles` registry (the canonical source for all role names in the system) and the `lt_config_role_escalations` table that defines which roles can escalate to which. `index.ts` exports role CRUD, escalation chain management, and RBAC-aware `canEscalateTo` checks; `sql.ts` contains all query templates.

The `LIST_ROLES_WITH_DETAILS` query joins three tables via CTEs (`lt_user_roles`, `lt_config_role_escalations`, `lt_config_roles`) against `lt_roles` — all LEFT JOINs, so it's safe but scales with the cross-product of roles × references. At typical cardinalities (tens of roles) this is negligible. The `deleteRole` function performs four sequential reference-count checks before deleting; these are intentionally non-transactional since the worst case (a concurrent insert between check and delete) is caught by FK constraints. The `replaceEscalationTargets` function uses an explicit transaction to atomically swap chain entries, preventing partial updates visible to concurrent readers.
